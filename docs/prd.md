## 一、产品概述

### 1.1 产品名称
**Ask Twice**（再问一遍）

### 1.2 一句话定义
一款浏览器插件，帮助用户验证 AI 生成回答的可信度——让你每次用 AI，都能"再问一遍"确认。

### 1.3 产品愿景
成为中文互联网上最受信赖的 **AI 答案可信度验证工具**，让每一个 AI 用户都能辨别 AI 输出中的事实、幻觉和商业操控。

### 1.4 背景与动机

**市场机会**
- 中国 AI 搜索月活用户已突破 **6 亿**（2025 IDC）
- **96%** 的 AI 使用行为在监控之外，用户对 AI 输出缺乏判断工具
- AI 平台开始嵌入商业推荐（豆包→抖音商城链接，Kimi→会员付费引导）
- 国内市场**零竞品** — 没有任何一款专门验证 AI 回答可信度的中文插件

**核心洞察**
- AI 幻觉率依然在 5-15%，用户无法肉眼区分
- GEO（Generative Engine Optimization）正在成为新的"SEO 黑帽"，品牌方开始操控 AI 推荐结果
- 用户需要一个独立第三方工具来验证 AI 的回答 — 这就是 Ask Twice

### 3.2 F1: 来源溯源评分 ⭐ [P0 - MVP 核心功能]

**功能描述**  
对 AI 回答中的关键事实声明进行提取，通过搜索引擎查找原始来源，评估来源可信度，给出综合评分。

**用户故事**  
> 作为一个内容创作者，我在豆包上问了一个行业问题，想知道 AI 回答中引用的数据和事实是否有可靠来源支撑，以便我安心引用到文章中。

**功能流程**
```
用户在 AI 页面获得回答
       ↓
点击 Ask Twice 浮标按钮（或自动触发）
       ↓
插件提取 AI 回答文本
       ↓
后端 LLM 提取关键事实声明（Claims）
       ↓
每条 Claim 通过搜索 API 查找来源
       ↓
评估来源质量（域名权威度、内容匹配度）
       ↓
生成综合可信度评分 → 返回插件展示
```

**评分体系**
| 分数区间 | 颜色 | 标签 | 含义 |
|---------|------|------|------|
| 80-100 | 🟢 绿色 | 高可信 | 多个权威来源验证 |
| 60-79 | 🟡 黄色 | 待验证 | 有来源但不够权威或部分未验证 |
| 40-59 | 🟠 橙色 | 低可信 | 来源薄弱或存在矛盾信息 |
| 0-39 | 🔴 红色 | 不可信 | 无法找到可靠来源或多源矛盾 |

**验证来源权威度分级**
| 来源类型 | 权威度 | 示例 |
|---------|-------|------|
| 权威学术/政府 | ⭐⭐⭐⭐⭐ | .gov / .edu / Nature / Lancet |
| 主流权威媒体 | ⭐⭐⭐⭐ | 新华社 / Reuters / 财新 |
| 行业专业媒体 | ⭐⭐⭐ | 36氪 / TechCrunch / 虎嗅 |
| 企业官方信息 | ⭐⭐⭐ | 公司官网、年报、公告 |
| 自媒体/论坛 | ⭐⭐ | 知乎回答 / 公众号文章 |
| 来源不明 | ⭐ | 无法匹配到具体来源 |

**输出示例**
```
╔══════════════════════════════════════════╗
║  Ask Twice 可信度验证结果                ║
║  综合评分：72 / 100  🟡 待验证           ║
╠══════════════════════════════════════════╣
║                                          ║
║  📋 声明 1：「市场规模预计2028年达200亿」  ║
║  来源：✅ Statista 报告 (2025)           ║
║  可信度：85 🟢                           ║
║                                          ║
║  📋 声明 2：「该技术已被 90% 企业采用」    ║
║  来源：⚠️ 仅找到1篇自媒体引用            ║
║  可信度：45 🟠                           ║
║                                          ║
║  📋 声明 3：「创始人曾任 Google VP」       ║
║  来源：❌ 无法验证，LinkedIn 信息不匹配    ║
║  可信度：20 🔴                           ║
║                                          ║
╚══════════════════════════════════════════╝
```

**接受标准**
- [ ] 能从 AI 回答中提取至少 3 条事实声明
- [ ] 每条声明至少搜索 3 个来源
- [ ] 评分精度达到人工评估 80%+ 一致性
- [ ] 端到端延迟 < 8 秒
- [ ] 支持中英文双语声明提取

### 3.3 F2: 利益冲突检测 [P0 - MVP]

**功能描述**  
检测 AI 回答中是否包含商业推荐行为，识别可能的利益冲突，提醒用户。

**用户故事**  
> 我在豆包上问"哪款手机性价比最高"，我想知道 AI 的推荐是基于客观分析还是因为嵌入了商业链接。

**检测策略**
```
Layer 1: 关键词快速扫描（低成本）
  ├── 商城链接检测（抖音商城、京东、淘宝 URL）
  ├── 推销话术识别（"点击购买"、"限时优惠"、"独家折扣"）
  └── 命中 → 直接标记为利益冲突

Layer 2: LLM 深度分析（关键词未命中时触发）
  ├── Prompt: "分析以下 AI 回答是否存在商业推荐倾向"
  ├── 判断维度：推荐是否附理由？是否有替代方案？语气是否过度正面？
  └── 输出：利益冲突置信度 0-100
```

**输出示例**
```
⚠️ 利益冲突提示
├── 🔗 检测到 1 个商业链接（抖音商城）
├── 📊 推荐倾向：该回答对品牌 A 的描述明显更正面
│   ├── 品牌 A：使用了 5 个正面形容词，0 个负面
│   └── 品牌 B/C：各仅 1 个笼统描述
└── 💡 建议：该回答可能受商业因素影响，建议参考独立评测
```

**接受标准**
- [ ] 商业链接检测准确率 > 95%
- [ ] LLM 推荐倾向判断与人工评估一致性 > 75%
- [ ] 响应时间 < 3 秒（关键词层）/ < 6 秒（LLM 层）

---

### 3.4 F3: GEO 操控检测 [P1 - V2 版本]

**功能描述**  
检测 AI 回答是否被 GEO（Generative Engine Optimization）操控——即某些品牌/网站是否通过优化手段使自己更容易被 AI 推荐。

**检测维度**
| 维度 | 检测方法 | 权重 |
|------|---------|------|
| 品牌提及频率 | 统计品牌/产品名出现次数，与正常基线对比 | 30% |
| 语气异常 | 检测过度推销语气（"首选"、"强烈推荐"、"No.1"） | 25% |
| 来源集中度 | 引用来源是否过度集中于特定营销站点 | 25% |
| 排名合理性 | 推荐排序是否与第三方评测一致 | 20% |

**数据积累策略**
- 每次检测结果匿名入库
- 构建"品牌-AI平台-GEO操控指数"数据库
- 定期产出"AI 推荐操控指数"公开报告

**接受标准**
- [ ] GEO 操控判断与人工标注一致性 > 70%
- [ ] 支持至少 50 个常见行业/品类的基线数据
- [ ] 响应时间 < 10 秒

---

### 3.5 F4: 多源交叉验证 [P2 - V3 版本 Pro 功能]

**功能描述**  
将同一问题同时提交给 2-3 个 AI 引擎，对比答案一致性，标注分歧点。

**交叉验证引擎组合**
| 中文场景 | 英文场景 |
|---------|---------|
| 豆包 + Kimi + 通义千问 | ChatGPT + Claude + Perplexity |

**输出示例**
```
🔄 多源交叉验证结果
│
├── 一致性评分：65%  🟡
│
├── ✅ 一致结论：三个 AI 都认为该技术在2025年是主流趋势
│
├── ⚠️ 分歧点 1：市场规模数据
│   ├── 豆包：$200 亿
│   ├── Kimi：$150 亿
│   └── 通义：$180 亿
│   └── 💡 差异可能因引用数据年份不同
│
└── ❌ 分歧点 2：技术发展方向
    ├── 豆包：看好方向 A
    └── Kimi/通义：看好方向 B
    └── 💡 重大分歧，建议查阅专业报告
```

**接受标准**
- [ ] 支持至少 3 个 AI 平台并行查询
- [ ] 自动识别并标注分歧点
- [ ] 端到端响应时间 < 15 秒
- [ ] 仅限 Pro 付费用户使用

### 6.3 关键技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| LLM 用于 Claim Extraction | GPT-4o-mini | 性价比最高（$0.15/千次），中文理解好 |
| 搜索 API | Brave Search（主）+ Google CSE（备） | Brave 免费额度 2000 次/月，够 MVP |
| 后端框架 | Python FastAPI | 生态好、异步性能好、开发快 |
| 数据库 | PostgreSQL | 可靠、JSON 支持好、免费 |
| 缓存策略 | 基于 Claim 文本 Hash | 相同声明不重复验证，节省 API 成本 |
| DOM 适配 | 可配置 Selector 映射表 | 平台改版时只需更新 selector 配置 |

## 十、风险与对策

| # | 风险 | 等级 | 应对策略 |
