三、核心技术方案
你要做的四个功能，技术路径：
① 来源溯源评分
提取 AI 回答中提到的信息点（claim extraction）
用搜索 API（Google/Bing/搜狗）查找原始来源
对来源网站做可信度评级（域名权威度、是否权威媒体/学术机构）
输出评分：每条 claim 的来源可信度 0-100
💡 可以用 LLM 做 claim extraction，成本关键是搜索 API 调用量
② GEO 优化痕迹检测
GEO（Generative Engine Optimization）是今年热门研究方向
Princeton 2024年论文首次系统研究了 GEO
检测方法：
统计分析：回答中品牌/产品提及频率是否异常
语言风格：是否有过度"推荐式"语气（"强烈推荐"、"首选"等）
交叉验证：同一问题问多个 AI，答案一致性
来源分析：引用来源是否集中于特定营销站点
💡 这是最有技术壁垒的部分，也是你的差异化
③ 利益冲突提示
检测 AI 回答中是否包含商品链接/推荐
判断推荐是否有商业动机（豆包嵌入抖音商城链接等）
可以维护一个"已知商业化 AI 推荐模式"数据库
💡 相对简单，规则引擎 + 轻量 ML 就能做
④ 多源交叉验证
同一问题同时查询 2-3 个 AI 引擎
对比答案一致性，标注分歧点
这个 Facticity.AI 已经在做类似的
💡 API 成本较高，建议做 Pro 付费功能
━━━━━━━━━━━━━━━━━━━━━
四、技术架构建议
用户浏览器
  │
  ├─ Chrome Extension (Content Script)
  │   ├─ 注入到 ChatGPT/豆包/Kimi 页面
  │   ├─ 监听 AI 回答 DOM 变化
  │   └─ 提取回答文本 → 发送到后端
  │
  └─ 后端 API (你的服务器)
      ├─ Claim Extraction (LLM)
      ├─ Source Verification (搜索API)
      ├─ GEO Detection (自研模型)
      ├─ Conflict Check (规则引擎)
      └─ 返回评分 → 插件 overlay 显示

关键技术选型：
前端：Chrome Extension Manifest V3
后端：Python/Node + LLM API
LLM：建议用 Claude/GPT-4o-mini（性价比）做 claim extraction
搜索：Google Custom Search API（$5/1000次）或 Brave Search API（免费2000次/月）
数据库：PostgreSQL + 来源可信度知识库

你说得对 — 自研模型是大厂/学术团队干的事，个人开发者应该站在巨人肩膀上，用现成的 LLM 能力来实现同等效果。

让我重新规划一下轻量化方案：

GEO 检测 → 不自研模型，用 LLM prompt 实现：
把 AI 回答丢给另一个 LLM（比如 Claude/GPT-4o-mini），prompt 设计成"分析这段回答是否有异常推荐倾向"
让 LLM 判断：品牌提及频率是否异常、语气是否过度推销、是否有可疑排名声明
本质上是 LLM-as-a-Judge，不需要训练任何模型
成本：GPT-4o-mini 大概 $0.15/1000次，完全可控
利益冲突检测 → 不用规则引擎，也用 LLM：
同样 prompt engineering："这段回答中是否包含商品推荐？推荐是否附带购买链接？是否有明显商业动机？"
加一个简单的关键词辅助（"点击购买"、"限时优惠"、商城域名列表）减少 LLM 调用
关键词命中 → 直接标记；不确定 → 再调 LLM 判断
整体架构简化成：
Chrome 插件 → 提取 AI 回答文本
↓
后端 API（一个 Python/Node 服务）
↓
┌─ Step 1: LLM 提取关键声明 (claim extraction)
├─ Step 2: 搜索 API 核查来源 (Brave/Google)
├─ Step 3: LLM 判断推荐倾向 + 利益冲突
└─ 汇总评分 → 返回插件显示

一个人完全能搞定的部分：
✅ Chrome 插件前端（Manifest V3 + Content Script）
✅ 后端 API（几百行代码）
✅ Prompt engineering（这是你的核心竞争力）
✅ 搜索 API 集成
不需要碰的：
❌ 训练模型
❌ 复杂规则引擎
❌ 大规模数据标注
核心壁垒从"自研模型"变成了"prompt 设计 + 产品体验" — 谁的 prompt 判断最准、UI 最好用、响应最快，谁就赢。
